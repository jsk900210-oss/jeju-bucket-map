# 제주버킷 리뷰맵 — AI 에이전트 작업 규칙

이 문서는 ChatGPT, Codex 및 기타 AI 에이전트가 `jeju-bucket-map` 저장소를 수정할 때 따라야 하는 공통 작업 규칙이다.

## 1. 브랜치 운영 원칙

* 실제 서비스 배포 브랜치는 `main`이다.
* ChatGPT와 Codex의 작업 브랜치는 `gpt`로 고정한다.
* AI 에이전트는 `main` 브랜치에 직접 커밋하거나 푸시하지 않는다.
* 모든 코드·문서 수정은 먼저 `gpt` 브랜치에 반영한다.
* `main` 반영은 사용자가 결과를 확인하고 승인한 뒤에만 진행한다.
* 사용자의 명확한 요청 없이 `gpt` 외의 추가 브랜치를 생성하지 않는다.
* 작업 시작 전 `gpt` 브랜치가 최신 `main`과 충돌하지 않는지 확인한다.

## 2. 한 에이전트는 하나의 작업만 수행

* 각 에이전트는 한 번에 정확히 하나의 작업만 맡는다.
* 여러 기능을 한 에이전트에게 동시에 맡기지 않는다.
* 작업 카드에 명시되지 않은 기능을 임의로 추가하지 않는다.
* 작업 중 다른 문제를 발견해도 현재 작업에서 함께 수정하지 않는다.
* 범위 밖 문제는 후속 작업 후보로만 기록한다.
* 검토 에이전트와 수정 에이전트의 역할을 분리한다.

예시:

```text
허용:
- CHANGELOG.md에 v21 기록 추가

금지:
- index.html 수정
- 디자인 변경
- 리뷰 저장 방식 변경
```

## 3. 작업 카드 필수 항목

모든 작업은 실행 전에 아래 항목이 정의되어야 한다.

```text
작업 ID:
담당 에이전트:
작업 상태:

단일 목표:
수정 허용 파일:
수정 금지 파일:
수정 금지 기능:

완료 조건:
검증 방법:
결과 증빙:
후속 작업 후보:
```

작업 범위가 불분명해도 에이전트가 임의로 확대 해석하지 않는다.

## 4. 작업 상태 표시

작업 상태는 다음 다섯 단계로 관리한다.

```text
⬜ 대기
🟨 진행 중
🟪 검토 중
🟥 수정 필요
🟩 완료
```

상태 변경 기준:

* 대기: 작업 카드만 작성된 상태
* 진행 중: 담당 에이전트가 작업을 시작한 상태
* 검토 중: 수정과 기본 검증이 끝난 상태
* 수정 필요: 오류 또는 완료 조건 미충족이 확인된 상태
* 완료: 사용자 확인 또는 승인까지 끝난 상태

## 5. 커밋 규칙

* 한 작업은 가능한 한 하나의 독립된 커밋으로 기록한다.
* 서로 다른 작업을 한 커밋에 섞지 않는다.
* 커밋 메시지에는 작업 ID를 포함한다.

커밋 메시지 형식:

```text
A01: add agent working rules
B03: migrate legacy review storage key
C02: extract storage logic
```

* 자동 생성 파일이나 불필요한 임시 파일을 커밋하지 않는다.
* 사용자가 승인하지 않은 변경을 `main`에 병합하지 않는다.

## 6. 수정 범위 제한

* 작업 카드에 지정된 파일만 수정한다.
* 목표와 관계없는 코드 정리, 이름 변경, 들여쓰기 정리는 하지 않는다.
* 기존 기능의 동작을 변경할 경우 영향 범위를 기록한다.
* 새로운 외부 서비스나 라이브러리를 임의로 추가하지 않는다.
* 비용이 발생할 수 있는 서비스는 사용자 승인 없이 연결하지 않는다.
* 개인정보 수집 기능을 임의로 추가하지 않는다.
* GitHub 토큰, API 키, 서비스키, 비밀번호를 코드나 문서에 저장하지 않는다.

## 7. 범위 밖 문제 기록 방법

현재 작업과 관계없는 문제를 발견하면 코드는 수정하지 않고 다음 형식으로 기록한다.

```text
발견 항목:
발견 위치:
예상 영향:
확인 근거:
권장 후속 작업:
긴급도:
```

후속 문제는 새로운 작업 ID를 부여한 뒤 별도 에이전트가 처리한다.

## 8. 에이전트별 역할 제한

### 기획 에이전트

* 작업 카드 하나만 작성한다.
* 직접 코드를 수정하지 않는다.

### 조사 에이전트

* 지정된 정보 하나만 조사한다.
* 공식 출처 또는 신뢰 가능한 1차 자료를 우선한다.
* 확인 날짜와 출처를 기록한다.
* 직접 코드를 수정하지 않는다.

### 코드 에이전트

* 지정된 기능 하나만 구현한다.
* 허용된 파일 외에는 수정하지 않는다.

### 디자인 에이전트

* 지정된 화면 또는 요소 하나만 수정한다.
* 데이터 처리나 서버 로직을 변경하지 않는다.

### 검토 에이전트

* 코드를 직접 수정하지 않는다.
* 문제 위치, 영향, 재현 방법, 수정 권고만 작성한다.

### 테스트 에이전트

* 지정된 테스트만 수행한다.
* 테스트 실패를 숨기거나 임의로 코드를 수정하지 않는다.

### 문서 에이전트

* 지정된 문서 하나만 작성하거나 갱신한다.
* 코드 동작을 변경하지 않는다.

## 9. 검증 규칙

### 문서 작업

* 버전, 날짜, 링크, 기능 설명이 실제 코드와 일치하는지 확인한다.
* 기존 기록이 의도치 않게 삭제되지 않았는지 비교한다.

### 코드 작업

* 정상 상황과 오류 상황을 모두 확인한다.
* 기존 핵심 기능이 유지되는지 확인한다.
* 브라우저 콘솔 오류 여부를 확인한다.

### 화면 작업

* 모바일 화면을 우선 확인한다.
* 변경 전과 변경 후 화면을 각각 남긴다.
* 글자 잘림, 버튼 크기, 터치 영역, 하단 시트, 지도 조작을 확인한다.

### 데이터 작업

* 장소명, 좌표, 노선, 요금, 운영시간은 공식 자료와 대조한다.
* 확인 날짜를 기록한다.
* 추정값을 확정 정보처럼 표시하지 않는다.

### 저장·인증 작업

* 저장 성공과 저장 실패 상황을 모두 확인한다.
* 브라우저 새로고침 후 데이터 유지 여부를 확인한다.
* 인증 우회 가능성을 확인한다.
* 개인정보와 비밀키가 노출되지 않는지 확인한다.

## 10. 작업 결과 보고 형식

각 에이전트는 작업이 끝난 뒤 아래 형식으로 결과를 남긴다.

```text
작업 ID:
담당 에이전트:
작업 상태:

완료한 내용:
수정한 파일:
수정하지 않은 범위:

검증 방법:
검증 결과:
실패하거나 확인하지 못한 사항:

화면 증빙:
발견한 후속 작업:
되돌리는 방법:
커밋 SHA:
```

확인하지 못한 사항은 완료한 것처럼 표현하지 않는다.

## 11. 화면 변경 증빙

화면이 달라지는 작업에는 다음 자료가 필요하다.

* 변경 전 화면
* 변경 후 화면
* 모바일 크기 화면
* 변경된 영역 표시
* 직접 확인한 기능 목록

시각적 변경이 없는 내부 로직 작업은 테스트 결과나 흐름도로 증빙한다.

## 12. main 병합 규칙

`gpt` 브랜치의 작업을 `main`에 반영하려면 다음 조건을 충족해야 한다.

1. 작업 카드의 완료 조건 충족
2. 검증 결과 기록
3. 주요 오류 없음
4. 사용자에게 변경 내용 설명
5. 사용자의 명확한 병합 승인

AI 에이전트는 사용자 승인 없이 `main`으로 병합하지 않는다.

## 13. 배포 후 확인

`main` 반영 후에는 다음 항목을 별도로 확인한다.

* GitHub Pages 빌드 성공 여부
* 라이브 주소 접속 여부
* 지도 타일 표시 여부
* 리뷰 폼 실행 여부
* 모바일 화면 표시 여부
* 브라우저 콘솔 오류 여부

배포 실패가 발생하면 기존 작업에 다른 수정을 섞지 않고 별도 복구 작업으로 분리한다.

## 14. 현재 저장소 기준

```text
저장소:
jsk900210-oss/jeju-bucket-map

배포 브랜치:
main

ChatGPT·Codex 작업 브랜치:
gpt

현재 운영 원칙:
gpt에서 작업 → 검증 → 사용자 확인 → 승인 후 main 반영
```
